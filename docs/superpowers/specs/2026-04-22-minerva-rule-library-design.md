# 规则库管理系统 — 技术设计（Design Spec）

- **成稿日期**：2026-04-22  
- **状态**：草案已通过对话评审，待读者最终批准后方可进入《实现计划》  
- **关联仓库**：`minerva`（当前同仓：`minerva-ui` 前端、Python API、独立 Worker 进程）  

---

## 1. 背景与目标

### 1.1 要解决的问题

构建一套**前后端分离**的**规则库管理系统**，在**多租户、租户下工作空间**隔离的前提下，对多类型规则进行**元数据、版本、发布、审计**等全生命周期管理；**设计态**以 **React Flow** 进行可视化编辑；**运行态**在服务端提供**可执行**的工作流/规则能力（分支、**部分动作/回调/异步**、**状态**、**重试**、**并发**），并采用 **Redis** 支撑**队列/锁/短期状态/延迟重试**，**PostgreSQL** 保存**长期权威数据与审计**。

### 1.2 需支持的规则/流程类型

- **文档/内容校审类**（禁忌词、格式、引用与可配置校验等）。  
- **业务/工作流类**（审批条件、if-then、与外部/内置流程**编排**的衔接，以本规格**白名单**与**集成**为准）。  
- **通用策略类**（**条件 + 动作** 或同等抽象，**不**在首版强绑单一 DSL 实现）。  
- 以上**多类并存**，采用**统一元数据模型 + 多类型规则扩展**。

### 1.3 非目标 / 不纳入本规格（首版不承诺，除非后文显式写「首版即」）

- 完整 **BPMN 标准**交换与**引擎替换**不承诺。  
- 企业级 **OIDC/SSO、MFA、跨地域多活** 等不默认纳入首版（可预留接口）。  
- **用户邀请/加入**已有组织可列为 **V2+**（本规格不强行展开，以免扩大首版范围）。  
- 旧版 **Internet Explorer** 支持（**Ant Design 6** 已不面向 IE）。

## 2. 总体架构

### 2.1 系统边界

- **Browser**：`minerva-ui`（**React 18+**、Vite、React Router）。  
- **API**：**FastAPI** 应用，负责**身份认证、租户/工作空间**作用域、**规则/版本/发布**的同步接口、**执行启停/查询/控制**（**不**在 API 进程内长时间运行业务长流程）。  
- **Worker**：**独立** Python 进程，从 **Redis** 取任务、持锁、**步进**执行、写 **PG** 审计与**里程碑**、使用 **延迟队列/重试**、发布**内部消息**以衔接 **MQ 消费**形态。  
- **PostgreSQL**：**定义域**与**可查询权威态**；**多租户/工作空间**、规则与版本、执行与**长期**审计。  
- **Redis**：**执行队列/延迟重试/分布式锁/短期执行机** 与**内部消息/流**；**不**作为唯一真相源替代 PG 对外**承诺**。  
- **External HTTP**：**动作**节点**默认** 通过 **HTTP** 与外部集成（超时、可配置成功判据）。  
- **External Data**：**轮询子任务** 节点**周期性**查询（**PG/Redis 自身状态**或**可配置**外部系统的**只读**查询，直至**终态**）。  
- **Message Queue 形态**（V1 基线，见 7.2）：在**不引入** 额外运维负担的前提下，**优先**以 **Redis 原生结构**（List/Stream 等 + **消费组/消费者**）实现 **「入站结果」与「出栈推进」**；若部署必须对接 **Kafka/Rabbit** 等，在实现阶段**抽象**为**可插拔实现**，本规格不强制 V1 完成全部适配。

### 2.2 多进程部署模型（同仓、二进程+数据层）

- **api**：`uvicorn` 或等效。  
- **worker**：Celery/ARQ/**自研**基于 Redis 的**消费者** 均可；首版在《实现计划》中**定稿**具体库，须满足**重试/延迟/可观测** 要求。  
- 二者**共享** `SQLAlchemy` 模型与**迁移**；**不**在 API 内跑长循环执行器。  

### 2.3 单体式演进边界

- 在领域包层面划分 `rules / execution / identity / infrastructure`，以便未来将 **执行服务** 拆为独立部署，**不**在首版拆服务。

## 4. 多租户、工作空间与成员

### 4.1 外显与实现命名

- **界面上**统一使用**「工作空间」**（**禁止** 与**「项目」**混称）。  
- **数据模型与 API** 以 **`Workspace` / `workspace_id`** 为准（表如 `workspaces`），**第二级**隔离为**租户 → 工作空间**；规则、执行、凭据的**主归属**为 **`workspace_id`**（在 `tenant` 上可**冗余**以加速**过滤**，但须**以关系一致**为约束）。  

### 4.2 首版产品约束（MVP 注册/登录）

- **注册**时：创建**平台用户**，并**同时**创建**新租户** + 一个**默认工作空间**；注册用户为该工作空间/租户的**Owner** 起点。  
- **加入** 已有租户/工作空间 的**邀请/审批** 不纳入首版**强制**交付，仅预留。  
- **鉴权**：短有效期 **JWT** 等（**Refresh** 与轮换策略在《实现计划》**定值**）。  
- **API** 对**所有**受保护端点执行 **`tenant` + `workspace` + 成员/角色** 校验；**不得**仅信任**客户端**传入 ID。  

### 4.3 数据域（概览，字段级在实现计划/迁移中展开）

- `users` | `tenants` | `workspaces` | `tenant_memberships` | `workspace_memberships` | `rules` | `rule_versions` | `workspace_credentials`（或 `integration_credentials`，**外键** 指向 `workspace`）| `executions` | `execution_events` 或**节点/里程碑**子表 等。  
- 凭据**不**在日志/审计中**明文凭据**重放。

## 5. 前端设计系统与信息架构

### 5.1 统一视觉与壳层

- **Button / Form / Menu**：以 **Ant Design 6** 为基，经 **主题 Token** 与**薄封装**（`AppButton`、`AppForm` 等）统一**变体**与**表单项** 错误、只读、帮助区。  
- **壳层**：`Header`（租户/工作空间/语言/主题/用户）、`Sider`（**菜单与路由**同步）、`Content` + **Breadcrumb**（`租户 / 工作空间 / 功能模块 / 子页` 等，**以路由元信息** 驱动 i18n）。  
- **React Flow** 区域：在 `Content` 内**可全高**；保证**与 AD 壳** 的**色板/边框** 协调。  
- **公开页**：**登录/注册** 使用**无**侧栏的**独立布局**；**受保护** 路由**包裹** 上述壳。  

### 5.2 API 基址

- **环境变量** 配置**后端**基地址（`VITE_API_BASE_URL` 等），**CORS** 在部署层与 FastAPI 层**成对**配置。

## 6. 规则资产与流程契约

### 6.1 存储

- 每条**发布/草稿**的图样以 **`RuleVersion.flow_json`** 存储 **React Flow** 的**可序列化** 导出。  
- **`flow_schema_version`**：与**校验器** 及 **Pydantic/JSON Schema** 对齐；**升级** 需**迁移**说明与**兼容**策略。  
- 解析后生成 **IR**（内部有向图，边/节点**类型**受**白名单**约束，见 7.1）。  

### 6.2 发布与执行前置条件

- **发布** 前必须通过**结构校验**与**可执行性** 检查；**非白名单** 节点/边 **不可** 发布。  
- 可选**模拟/干跑** 为**首版可选项**（若纳入《实现计划》，须**不**对生产外部系统产生**副作用** 或**显式** sandbox）。

## 7. 执行域：IR、轮询、消息队列、Redis 与 PG

### 7.1 自研图 IR 与**节点白名单**（首版必须显式列清单于《实现计划》）

- **原则**：**React Flow JSON → 校验 → IR**；**解释器** 在 **Worker** 中**单执行实例** 步进。  
- **有环/最大步数/超时** 的**处理** 在《实现计划》**必须** 写清；**无限制** 循环**不得** 默认可达。  
- **锁**：**以 `execution_id` 为粒度** 保证**同一** 执行**的** 推进**互斥。  
- **幂等**：`Executions` 上 **`idempotency_key`** 防**重复** 触发；**与动作** 侧**幂等** 联合策略在**集成** 章节展开。  

### 7.2 总原则与两种**异步/等待** 形态**并存**（V1 允许）

1. **轮询子任务/状态 节点**  
   - Worker 按**可配置** **间隔/退避/超时** 轮询，直至 **PG 中** 或**外部只读** 查询返回**终态**；**每轮** 写 **执行里程碑** 到 **PG**（**权威** 查询）。短周期**工作数据** 可落在 **Redis**，但**以 PG 对外展示** 为准。  

2. **消息队列 生产-消费 形态**  
   - 某**节点** 将**结构化结果/事件** **写入** **约定** 的**消息载体**（**V1 基线** 建议 **复用同一 Redis 集群** 的 **Stream/List + Consumer**；**不** 在首版**强制** 引入**新的** 中间件种类）。  
   - **同仓 Worker** 或**同消费者池** 从队列**取消息**，按 **`execution_id` + 节点/游标** **恢复/推进** 解释器；**与** 轮询 形态**可** 在同一张规则图中**同时出现**（**不同** 边与节点**类型** 区分**语义**）。  
   - **安全**：**内部队列** 须带**防伪造** 与**租户/工作空间** 前缀/命名**隔离**；**对外** 若暴露**回调/入口**，须带 **token/签名** 与**幂等** 约定（**实现** 在计划中）。  

### 7.3 权威与一致性

- **对外**「执行/节点**状态/历史**」的**展示** 以 **PG** 为**可依赖** 来源。  
- **Redis** 为**执行机** 与**队列的** 短期、**可丢失后重建** 的数据（**须** 能从 **PG+规则** 或**明确** 的**可恢复** 点**重建**），但**不** 与 **PG** 长期矛盾；**冲突** 时**以 PG 为最终** 对账/修复策略在**运维** 小节定义。  

### 7.4 重试、死信与可观测

- **指数退避/最大重试/死信/人工** 在《实现计划》 中**以数值或默认** 给出；**至少** 提供**失败原因** 与**关键** 时间戳/节点 ID 的**审计** 字段。  
- 日志**不得** 打印**密钥/令牌** 明文。  

---

## 8. API 与错误

- 业务**错误** 返回**稳定** `code` + **可参数化** 字段；**人类文案** 以**前端** i18n 翻译。  
- **HTTP** 与**鉴权/租户/工作空间** 错误使用**可枚举** 的**分类**，防止**越权** 下**信息泄露**（如**统一 404/403 策略** 在**实现** 中**统一**）。  

---

## 9. 安全基线

- 密码**哈希**（Argon2 或 **bcrypt**），**不** 存明文。  
- **限流**（登录/注册/敏感写）。  
- **CORS/HTTPS** 部署约定。  
- **多租户/工作空间** 全链路**过滤** 与**集成测试** 覆盖。  

---

## 10. 风险与分阶段

- **高**：首版 同时 包含 **管理+执行+多租户+双异步形态+Redis+PG+AntD6+ReactFlow**，需《实现计划》 明确**里程碑** 与**最小可跑** 节点/动作 集合，避免范围失控。  
- **中**：**自研** IR/解释器/锁/对账 的**边界情况** 需**单元+集成+并发** 测试。
